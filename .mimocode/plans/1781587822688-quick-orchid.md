# Bug 检查报告 — SnapVision Desktop

## 发现的问题（按严重程度排序）

---

### Bug 1: `compressImage` 缺少 `onerror` 处理 — 可导致应用挂起

**文件**: `src/App.tsx:23-36`

**问题**: `compressImage()` 创建 `new Image()` 后只监听了 `onload`，没有 `onerror`。如果图片数据损坏或格式不支持，Promise 永远不会 resolve，`runRecognition` 将永远卡在 loading 状态，用户无法操作。

```ts
// 当前代码
img.onload = () => { ... resolve(...) }
// 缺少: img.onerror = () => reject(new Error('图片加载失败'))
```

**修复方案**: 添加 `img.onerror` 回调 reject Promise。

---

### Bug 2: 缺少顶层 ErrorBoundary — 渲染错误导致白屏

**文件**: `src/main.tsx`

**问题**: `main.tsx` 中 `<App />` 没有被 `<ErrorBoundary>` 包裹。`ErrorBoundary` 组件已存在于 `src/components/ErrorBoundary.tsx`，但未在顶层使用。任何组件渲染错误都会导致整个应用白屏崩溃，无恢复能力。

**修复方案**: 在 `main.tsx` 中用 `ErrorBoundary` 包裹 `<App />`。

---

### Bug 3: `runRecognition` 无并发控制 — 快速重复触发导致状态混乱

**文件**: `src/App.tsx:203-263`

**问题**: `runRecognition` 没有取消前一次请求的机制。如果用户快速触发两次识别（如快速拖入新图片），两次异步操作并行执行，后完成的会覆盖先完成的状态，导致显示的数据与实际请求不一致。

**修复方案**: 使用 `useRef` 持带递增的 request ID，在 Promise resolve 时检查是否仍然是最新请求，如果不是则丢弃结果。

---

### Bug 4: `stderrBuf` 无限增长 — 长时间运行内存泄漏

**文件**: `electron/main.ts:387-399`

**问题**: `stderrBuf` 在 OCR 进程运行期间持续追加 stderr 输出，永不清理。长时间运行的应用（如数小时不重启）可能导致此缓冲区占用大量内存。

**修复方案**: 在 `stderrBuf` 超过一定大小（如 10KB）时截断，只保留尾部内容。

---

### Bug 5: `app.on('before-quit')` / `window-all-closed` 的 async handler 不被 Electron 等待

**文件**: `electron/main.ts:1087-1093`

**问题**: `app.on('before-quit', async () => { await stopOcrServer() })` 和 `app.on('window-all-closed', async () => { await stopOcrServer(); ... })` 使用了 async 回调，但 Electron 不会 await 这些 handler。`stopOcrServer` 可能未完成进程就被系统回收。

**修复方案**: 改为同步方式注册 quit 保护——在 `before-quit` 中仅发送 SIGTERM 并设置一个短 timeout，不依赖 await。或者利用 Electron 的 `will-quit` 事件（它会阻塞退出）。

---

### Bug 6: `electron.d.ts` 中 `StockParseResult` 接口字段重复声明

**文件**: `src/types/electron.d.ts:76-86` 和 `122-142`

**问题**: `raw_texts`, `filtered_texts`, `ignored_texts`, `matched_stock_code` 等字段在 `StockParseResult` 中被声明了两次。虽然 TypeScript 允许（后声明覆盖前声明），但这是明显的复制粘贴错误，可能导致维护困惑。

**修复方案**: 删除重复的声明块（第二个，即 122-142 行）。

---

## 不修改的问题（设计层面，非 Bug）

- `ocr:retry` 中 kill 后立即 start 的端口竞争 — 已由 `killStaleProcess` 处理
- `net.request` fetch-quote 不检查 HTTP 状态码 — 已通过 regex match fallback 处理
- Flask `_history_service` 单例无线程安全 — dev 模式单线程运行，无风险

## 验证方式

1. Bug 1: 尝试拖入一个非图片文件（如 .txt），确认不会永久 loading
2. Bug 2: 临时在某个组件中 `throw new Error('test')`，确认 ErrorBoundary 捕获而非白屏
3. Bug 3: 快速连续拖入两张图片，确认只显示最后一张的结果
4. Bug 4: 监控长时间运行后的内存使用
5. Bug 5: 启动后立即 Cmd+Q 退出，检查 OCR 进程是否被清理
6. Bug 6: `npm run build` 确认 TypeScript 编译无 warning
